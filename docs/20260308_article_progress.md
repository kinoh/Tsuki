## ここまでに固まった内容

### プロンプトとハーネス
- プロンプトは人格の根幹として扱う。
- 「〇〇っぽく」といった書き方は簡単だが、可塑性を失わせ、自己認識を平坦化しやすい。
- ハーネスは品質を自動的に担保し、開発サイクルの手間を決定的に変える。
- 会話ログを使って「人格のレグレッションテスト」ができる点も重要。
- 主例は `chitchat` にする。
- `core-rust/tests/integration/scenarios/chitchat.yaml` と result の断片をそのまま載せる。
- `{{conversation_sample}}` は judge が人格比較に使う基準会話のプレースホルダとして説明する。
- `fuzzy_concept_intro_query` と `shell_exec_news_fetch` は軽く触れる程度にする。
- `fuzzy_concept_intro_query` では、モデル変更で起きた過剰な情報提供や、曖昧な問いをミニ講義にしてしまう傾向を例示する。
- `shell_exec_news_fetch` では、ツール使用は tool description やエージェントのプロンプトも関与して制御しにくいことを踏まえ、実用的な使い方ができているかを見る例として触れる。

### 概念グラフ
- 関係そのものを記述して自然に連想を扱えることを主軸に書く。
- 人間の想念は神経ネットワークの発火パターンとしてノードに抽象化できるはず、という見方を下敷きにする。
- 「行為」も例外ではなく、概念グラフ上で表現できるものとして捉える。
- その結果として、概念グラフ上の表現から動的ロードへつながる実装が自然で、美しい設計だと考えている。
- 例示としては、概念グラフツールのインターフェースがどのようなものか、router がどのような出力をしているかを示せばよい。

### イベント駆動
- Tsuki の思考は、ひとつづきの直列系列として扱う。
- `core-rust` ではその表現としてイベントストリームを明示的に実装している。
- `Mastra` のスレッドモデルはこの感覚とやや噛み合わず、移行の動機のひとつになった。
- LLM の逐次的なトークン生成は入力や直前文脈に引きずられやすく、そのままでは単線的すぎる、という問題意識から入る。
- それに対して、内部では分散的かつ非同期な反応を持ちつつ、表層では直列的な思考として経験される構成を目指す。
- `submodule` は、その分散した思考傾向を担う小さな役割単位としてここで説明する。
- `decision` はイベント列そのものではなく、その中からいま表層に出すものを選び取る役割として書く。
- 例示はログ断片ではなく `mermaid` でイベント伝達を示す。
- 図では `router -> submodules -> decision` の連鎖は点線で示しつつ、これは同期的な call graph ではなく、各モジュールは他のモジュールへの約束を持たないことを明記する。
